Open Air Foundation · Hackathon Build Spec

Interactive map for air-quality monitoring density & the inequality it reveals

A single web tool that overlays where air-quality monitors actually exist on top of where air pollution kills the most people — making the monitoring gap impossible to ignore, and downloadable for anyone who wants the data.

Nuxt 3 + Leaflet NestJS API PostgreSQL + PostGIS Nginx · Docker Hackathon MVP → Production roadmap
01 · Overview & Goal

The one-sentence goal

Combine the AirGradient monitor map (all public monitors — low-cost & reference, every manufacturer) with State of Global Air health data (deaths attributable to air pollution per country/region) so a viewer can instantly see that the places suffering the most are often the places measuring the least.

Plain-language version — for non-developers Think of it as two maps laid on top of each other. One shows little dots for every air-quality sensor in the world. The other shades each country by how many people die from dirty air. Where there are many deaths but few dots, that's the inequality we want to highlight — and the call to put more monitors there.

What this document is

A complete build specification: the technology choices, how data moves through the system, the database design, the API, how it scales from a hackathon demo to a global production service, and a day-by-day work plan. It is written to be handed directly to Claude Code (or any engineer) to start implementation, while staying readable for product and foundation stakeholders.

Primary outcome

Make the gap striking

Visually pair pollution health burden with monitor density per capita, country by country.

For the foundation

A campaign asset

Embeddable on the foundation website, with a guided "story mode" highlighting under-monitored regions.

For everyone

Open & downloadable

Filter the data live and export any view to CSV / Excel — monitors, counts, per-capita density.

02 · Product Scope

What the map shows & does

Derived directly from the hackathon brief. These are the user-facing capabilities the architecture must support.

Map layer

All publicly available monitors

Low-cost and reference stations from every manufacturer, drawn from the new AirGradient Map API. Coloured by current AQI; clustered at low zoom.

Density layer

Monitor count — absolute & per capita

Per country and region, so coverage density is comparable between, say, Germany/Europe and places with almost none.

Context panel

Current & average air pollution + health impact

For any selected area: live readings, long-run averages, the death toll attributable to air pollution, and an interactive yearly deaths-trend chart (1990–2023) you can hover for each year's value.

Filters

Slice the data

By monitor type, manufacturer, and online/offline status.

Export

Download what you see

Any filtered view → CSV or Excel: monitor list, counts, per-capita figures, and health metrics.

Narrative

"Inequality" story mode

A scripted sequence: show worst pollution → fade to black → reveal sparse monitor density. Built for a website hero video.

Gap levels

5-level monitoring gap

Each country gets a blue→red level from gap = deaths/100k ÷ monitors/100k, ranked as quintiles among countries that have monitors.

Investment

Monitors needed & cost

Per country: how many more monitors to reach the good zone, with low-cost vs reference cost and a donation-needed bar.

Worked example — Berlin (from the brief) Click Berlin and the panel answers four questions: (1) every public monitor shown, (2) monitor count in Germany and per capita vs. Europe, (3) current & average pollution in Berlin from real readings, (4) the health impact for Germany — e.g. deaths attributable to air pollution.
03 · Technology Stack

The stack

Chosen for speed-to-demo at the hackathon while being a real foundation you can grow. TypeScript end-to-end so one language spans frontend, backend, and shared data models.

Frontend
Nuxt 3 (Vue 3)
+ Leaflet · Pinia · Tailwind · TypeScript

Server-side rendering for fast first paint and good SEO on the foundation site. Leaflet for the map — light, free, OpenStreetMap tiles, mature clustering & choropleth plugins. Pinia holds filter state; ECharts/Chart.js renders the health charts.

Backend / API
NestJS
Node + TypeScript · REST (OpenAPI)

Structured, modular framework — controllers, services, DI, validation, Swagger out of the box. Hosts the public API, the ingestion scheduler, and the export service as clean separate modules.

Database
PostgreSQL + PostGIS
+ TimescaleDB (measurements)

PostGIS gives real geospatial queries — "monitors in this bounding box", point-in-country, per-area counts. TimescaleDB turns the readings table into an efficient time-series store as data grows.

Cache / Jobs
Redis + BullMQ
caching · rate-limit · job queue

Caches hot endpoints (the global view, density rankings), backs the rate limiter, and runs the scheduled ingestion jobs that poll the external data sources.

Edge / Infra
Nginx + Docker
reverse proxy · TLS · static + tiles

Nginx terminates TLS, serves the built Nuxt app and static assets, and reverse-proxies /api to NestJS. Everything ships as Docker containers via Docker Compose — one command to run the whole system.

Tooling
Prisma · Zod · pnpm
ORM · shared schemas · monorepo

Prisma (or TypeORM) for typed DB access & migrations; Zod schemas shared between API and frontend in a /packages/shared workspace so types never drift.

Why these and not the trendy alternatives Leaflet over Mapbox: free & no token limits for a public foundation site. Postgres+PostGIS over a NoSQL store: the core questions are geospatial & relational (counts per area, joins to health data). NestJS over a bare Express app: structure that survives past the hackathon when more people touch the code.

05 · Data Flow & Pipeline

How data moves through the system

Two different rhythms: monitor readings change constantly and are polled often; the roster of stations and the health datasets change rarely and are synced slowly. Each external source has its own job with its own schedule.

Pull live readings & monitor roster from AirGradient

A scheduled worker calls the AirGradient Map API. New stations are upserted into monitors; the latest values become rows in measurements. Each monitor is tagged with type, manufacturer, owner, and the OAF-donated flag if present.

readings ~ every 10 min · roster ~ daily

Import health-impact data (State of Global Air)

The GBD-based death & DALY figures are loaded as a versioned batch into health_impacts, keyed by country, year, and pollutant. This is reference data — refreshed only when a new annual release ships.

batch · on new release

Load geographies & population once

Country and region boundaries (PostGIS geometries) plus population figures are seeded so we can do point-in-area lookups and compute per-capita density. Urban-center polygons from the API enrich the city level.

seed · annual refresh

Geo-tag & aggregate

Each monitor's point is matched to its country / region / urban center via PostGIS. A job then recomputes counts — total, online, reference vs. low-cost, and per 100k people — and writes them to density_stats (and matching materialized views).

after each sync

Serve via API, cached in Redis

The frontend requests only finished answers: GeoJSON for the map viewport, a density object for a country, a health series for a chart. Hot responses are cached in Redis with short TTLs and busted when a sync completes.

real-time · TTL 1–10 min

Render & export in the browser

Nuxt + Leaflet draws clustered monitors and a choropleth health overlay; the side panel shows the selected area. The same filtered query can be streamed straight to CSV / XLSX by the export service.

on demand

Reliability rules baked into the pipeline

  • Idempotent upserts — re-running a sync never duplicates monitors (keyed on source + external_id).
  • Every run is logged in an ingestion_runs audit table: source, started/finished, rows touched, status — so a stalled feed is visible.
  • Graceful staleness — if an external source is down, the last good data stays served; monitors carry a last_seen so the UI can mark them offline.
  • Backfill-friendly — jobs accept a date range so historical readings can be re-imported without touching live logic.

Relationships — regions 1—∞ countries 1—∞ urban_centers · countries/urban_centers 1—∞ monitors · monitors 1—∞ measurements (time-series) · countries 1—∞ health_impacts · density_stats & ingestion_runs are derived/audit tables, not edited by hand.

Why a separate aggregate table Counting monitors per country across the whole world on every map move would be slow and would hammer the database. density_stats stores those answers already computed, refreshed by a background job — the API just reads one indexed row. This is the single most important decision for keeping the map fast at global scale.
07 · API Design

Public REST API — /api/v1

Read-heavy and stateless. Every endpoint returns JSON (GeoJSON where geographic), is cacheable, and maps to a screen feature. Auto-documented with Swagger/OpenAPI from NestJS decorators.

Monitors & map
GET
/monitors?bbox=&type=&manufacturer=&oaf=&status=

Monitors in a viewport as a GeoJSON FeatureCollection, server-clustered at low zoom. The map's primary call.

GET
/monitors/:id

Full detail for one monitor: metadata, owner, OAF flag, latest reading.

GET
/monitors/:id/measurements?from=&to=&interval=

Time-series readings for the history chart (raw or bucketed averages).

Density (the inequality numbers)
GET
/density?scope=country&iso3=DE

Counts & per-capita density for one area — drives the side panel (the "Berlin / Germany" answer).

GET
/density/ranking?metric=per_100k&order=asc&limit=

Most under-monitored areas — powers story mode and the "where it's needed most" view.

GET
/density/choropleth?metric=&scope=

One value per geography for shading the map overlay.

Health impact
GET
/health-impacts?iso3=DE&pollutant=pm25&metric=deaths

State of Global Air figures for an area — death toll & rates for the context panel. Omit year to get the full yearly series for the trend chart.

GET
/health-impacts/compare?iso3=DE&vs=region

Country vs. its region / world — the comparison that makes inequality legible.

Geography & meta
GET
/countries

/regions & /urban-centers — lookup lists & boundaries for filters and overlays.

GET
/stats/global

Summary cards: total monitors online, countries covered, % population under-monitored.

Export
GET
/export?format=csv|xlsx&dataset=monitors|density&…filters

Streams the current filtered view as a downloadable file — same filters as the map.

Conventions

  • Versioned under /api/v1; breaking changes go to /v2.
  • Cacheable — GETs send Cache-Control + ETag; Redis mirrors heavy ones server-side.
  • Validated — query params parsed by Zod/DTOs; bad input returns a clear 400.
  • Rate-limited per IP at Nginx + Redis so the public endpoint can't be abused.
08 · Data Export (CSV)

Download what you see — not just one file

Here's the decision on the export feature, made for you: rather than a single dump, the tool offers five focused exports — each answering a different question — with the monitoring-gap dataset as the headline.

The decision — why five exports, not one Different users need different grains: researchers want raw per-monitor rows, while journalists / policymakers want one tidy row per country to drop into a slide. So exports are split by level of detail instead of one unwieldy mega-file.

How it works

Same filters

What you see is what you get

The download button sends the exact same filters as the map (type, OAF, status, area) to /export — the file always matches the on-screen view.

UTF-8 + Excel

Thai-safe encoding

CSV is UTF-8 with a BOM so Excel opens Thai & other scripts correctly. An .xlsx option is available via format=xlsx.

Streamed

Scales to big files

Rows stream from the DB row-by-row, so tens of thousands of rows export without blowing up memory; large files can be parked on object storage.

Note for Claude Code Extend the existing /export with a dataset=gap|monitors|density|health|measurements param — no new endpoint needed. The gap dataset joins density_stats × health_impacts on country and computes gap_ratio in the service layer.
09 · Scaling Plan

From one demo box to a global service

The same codebase runs at all three phases — only the deployment topology changes. Build for Phase 0 at the hackathon, but make none of the choices that block Phase 2.

Phase 0 · Hackathon

Single host

~1 VPS · hundreds of users
  • Docker Compose: Nuxt, NestJS, Postgres+PostGIS, Redis, Nginx on one machine
  • OpenStreetMap raster tiles
  • Precomputed density_stats + aggressive Redis cache
  • One worker process for ingestion jobs
Phase 1 · Pilot

Split & managed

managed DB + CDN · 10k+ users
  • Managed PostgreSQL (read replica) + TimescaleDB for readings
  • Cloudflare CDN in front of tiles, static & cached API
  • API scaled to 2–3 stateless instances behind the proxy
  • Workers split onto their own service; object storage (S3) for exports & raw dumps
Phase 2 · Global

Elastic & observable

Kubernetes · autoscaling · global
  • K8s with horizontal autoscaling on API & workers
  • Precomputed vector tiles for monitors — millions of points stay smooth
  • DB partitioning + replicas; materialized views refreshed on schedule
  • Observability: Prometheus + Grafana, Sentry, structured logs; CI/CD pipeline

The four levers that actually matter for scale

Lever 01

Precompute

Aggregates & materialized views, never live counts.

Lever 02

Cache & CDN

Redis + Cloudflare absorb repeat reads at the edge.

Lever 03

Tile the points

Server clustering → vector tiles for huge monitor counts.

Lever 04

Stateless API

No session in memory → add instances freely.
